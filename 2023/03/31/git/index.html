8、查看状态

git status命令用于显示工作目录和暂存区的状态。使用此命令能看到那些修改被暂存到了, 哪些没有, 哪些文件没有被Git tracked到。 git status不显示已经commit到项目历史中去的信息。看项目历史的信息要使用git log.

总结：git status 命令用于查看在你上次提交之后是否有对文件进行再次修改。

git开发错分支

代码未提交时：

  1. git add . (把所有改动暂存)
  2. git stash (把暂存的文件提交到git的暂存栈)
  3. git checkout 本该提交代码的分支
  4. git stash pop (将暂存栈中的代码放出来)

代码提交了

  1. git checkout 不该提交代码提交了代码的分支
  2. git reset HEAD~1 （最近一次提交放回暂存区, 并取消此次提交）
  3. git stash (把暂存的文件提交到git的暂存栈)
  4. git checkout 应该提交代码的分支
  5. git stash pop
  6. git checkout 不该提交代码提交了代码的分支 （等你把代码提交到了正确的分支后，再次切到刚刚错的分支）
  7. git push origin 错误的分支 -f (把不该上去的文件回退掉)

版本回退

git reset

reset命令可以添加很多参数，常用的有--mixed、--soft和--hard三种

三个参数大体上的区别为：

  • --mixed：为默认值，等同于git reset。作用为：将文件回退到工作区，此时会保留工作区中的文件，但会丢弃暂存区中的文件；
  • --soft：作用为：将文件回退到暂存区，此时会保留工作区和暂存区中的文件；
  • --hard：作用为：将文件回退到修改前，此时会丢弃工作区和暂存区中的文件；

git revert

revert是回滚，重做的意思。不同于reset直接通过改变分支指向来进行版本回退，并且不产生新的提交；revert是通过额外创建一次提交，来取消分支上指定的某次提交的方式，来实现版本回退的

适用场景： 如果我们想撤销之前的某一版本，但是又想保留该目标版本后面的版本，记录下这整个版本变动流程，就可以用这种方法。

  2. 提交: 使用”git commit -m ‘’”，此时可以用”git log”查看本地的版本信息，可见多生成了一个新的版本
  3. 使用”git push”推上远程库

revert与reset的选择

由于checkout会造成游离的提交对象链，所以，一般不使用checkout而是使用reset和revert进行版本回退：

  • revert通过创建一个新提交的方式来撤销某次操作，该操作之前和之后的提交记录都会被保留，并且会将该撤销操作作为最新的提交；
  • reset是通过改变HEAD和分支指针指向的方式，进行版本回退，该操作之后的提交记录不会被保留，并且不会创建新的提交；

在个人开发上，建议使用reset；但是在团队开发中建议使用revert，特别是公共的分支（比如master)，这样能够完整保留提交历史，方便回溯。

方法 效果 向前回退 向后回退 同步修改HEAD与分支指向
git reset –hard HEAD^ 往前回退1次提交 能 否 是
git reset –hard HEAD~n 往前回退n次提交 能 否 是
git reset –hard 回退到指定commit id的提交 能 是 是
git revert HEAD 重做最新一次提交 能 否 是
git revert HEAD^ 重做倒数第二次提交 能 否 是
git revert HEAD~n 重做倒数第n+1次提交 能 否 是

git stash

git stash指令的作用为：对没有提交到版本库的，位于工作区或暂存区中游离的修改进行保存，在需要时可进行恢复。

当在develop分支上开发新功能的时候，master分支出现紧急情况需要切换回去进行修复。但是，当前分支的新功能还没开发完全，贸然切换分支，原来开发的内容就会因被覆盖而丢失，怎么办呢？

git stash可以将当前dev分支上，位于在工作区或暂存区中的修改，在未提交的情况下进行了保存；并且将分支回退到修改前的状态，保存过后，就可以很顺畅地切换回master分支了。

git merge和git rebase的区别

  1. git merge：
    • git merge 是将一个分支的更改合并到另一个分支中。合并后，会生成一个新的合并提交（merge commit）来将两个分支的历史记录合并在一起。
    • 这种合并方式保留了原始分支的完整历史记录，每次合并都会创建一个新的合并提交，形成一个分支的分支历史。
    • 合并后的分支结构会变得更清晰，但也会有较多的合并提交，可能会导致历史记录较为混乱。
  2. git rebase：
    • git rebase 是将一个分支的更改以衍合（cherry-pick）的方式逐个应用到另一个分支上，而不会创建新的合并提交。这会改变提交的顺序，使其看起来好像是在另一个分支上直接提交的。
    • 这种合并方式会使分支历史变得更线性和整洁，因为不会创建多余的合并提交。
    • 但需要谨慎使用 git rebase，因为它会改变提交的历史记录，可能会导致其他开发者在共享分支上的工作产生冲突。

关键区别总结如下：

  • git merge 会创建合并提交，保留完整的分支历史，分支结构更清晰，但历史记录可能较复杂。
  • git rebase 不会创建新的合并提交，而是改变提交的顺序，使历史记录更线性，但可能会改变提交历史，因此需要小心使用。
